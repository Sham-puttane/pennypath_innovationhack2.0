# PennyPath — Handover to Person 1 (Frontend)

## What's Done

### Person 2 (Sham-puttane) — Data Ingestion & Knowledge Graph

**Data Pipeline** (`backend/ingest.py`):
- Downloaded and extracted text from 6 real financial data sources
- 4 PDFs: CFPB Your Money Your Goals toolkit (split into 9 modules), CFPB Behind on Bills, CFPB Debt booklet, FEMA EFFAK
- 12 State Farm Simple Insights articles (individually scraped)
- 2 CFPB Newcomer's Guides (scraped)
- Total: 26 text files, 634K characters in `backend/input/`

**Knowledge Graph** (`backend/build_graph.py`):
- Used GraphRAG + Gemini 2.5 Flash to extract entities from all 26 documents
- Built graph: 4,194 entities, 8,066 edges, 21 communities
- Entity types: FINANCIAL_CONCEPT, ACTION, FINANCIAL_PRODUCT, ORGANIZATION, RISK, GEO, PERSON, EVENT
- Community detection via Louvain algorithm

**Community Reports** (`backend/generate_reports.py`):
- 21 narrative summaries (one per community), generated by Gemini

**Entity Embeddings** (`backend/generate_embeddings.py`):
- 4,194 entity embeddings (3072-dim) for Q6 semantic search
- Stored in `backend/output/entity_embeddings.npz` (gitignored, regenerable in ~8 min)

**Query API** (`backend/query_graph.py`):
- `traverse_from_profile(q1, q2, q3, q4, q5, q6)` — maps intake answers to graph entry nodes, BFS traversal, returns typed nodes
- `search(query)` — semantic search using embeddings (for Q6 free text)
- `get_community_info(id)` — community details with report
- `get_node_neighborhood(name)` — click-to-expand on graph
- `get_graph_for_visualization(nodes, edges)` — vis.js-ready format
- Persona detection is graph-driven: checks which corpus entity clusters (IMMIGRANTS, STUDENT LOANS, PREPAID CARD, etc.) the traversal touches

### Person 3 (mschinmayi07) — FOO Engine, Personalization, Server

**FOO Rule Engine** (`backend/foo_engine.py`):
All 10 rules from the brief implemented:

| Rule | What It Does |
|------|-------------|
| 1 | No bank account = always Step 1 |
| 2 | High-APR debt (>15%) = pay before saving |
| 3 | Savings < $500 = $500 buffer BEFORE debt attack (exception to Rule 2) |
| 4 | Insurance gap = State Farm card AFTER income stabilizes. Crisis = Step 6-7, Stable = Step 2-3 |
| 5 | Gig/cash income = emergency fund is $1,000 not $500 |
| 6 | No credit history = secured card step after bank account exists |
| 7 | Student loans = IDR check before standard repayment |
| 8 | Employer 401k match = catch it before paying extra debt |
| 9 | Multiple debts = avalanche method (highest APR first) |
| 10 | Savings > $1k + no high-APR debt = investment steps unlock |

**State Farm Card**:
- Embedded as a numbered step (not a banner)
- Position is dynamic: stable user with gap = Step 2-3, crisis = Step 6-7, no income yet = deferred
- Card text includes actual savings amount: "You have under $500 saved. One break-in could wipe this out..."
- If user already has all insurance = card does not appear

**Gemini Personalization** (`backend/personalize.py`):
- Single LLM call per session, runs AFTER FOO ordering
- Does NOT change order, add, or remove steps
- Rewrites action labels into warm, personal language
- Echoes user's Q6 words when relevant
- Falls back to original text if API keys are rate-limited

**Flask Server** (`backend/server.py`):
- `POST /api/query` — full pipeline (graph traversal -> FOO -> personalization)
- `POST /api/whatif` — What-If slider (re-runs FOO only, ~50ms, no GraphRAG re-query)
- `POST /api/search` — semantic search for Q6
- `GET /api/community/<id>` — community info with report
- `GET /api/node/<name>` — node neighborhood (click-to-expand)
- `GET /api/health` — health check
- `GET /api/personas` — 3 demo personas (Maria, James, Aisha)

**What-If Simulator** (6 scenarios per brief):
- lose_job, earn_more, pay_off_debt, get_renters_insurance, open_bank_account, get_steady_job
- Session cache stores graph traversal; slider only re-runs FOO engine
- No Gemini call on slider change (per brief)

**Personas** (graph-derived, 8 + general fallback):
- international_student, recent_immigrant, gig_worker, single_parent, first_gen_college_student, unbanked, veteran, reentry, general

# Install dependencies

## What Person 1 Should Build

### Screen 1: Intake Form
- 5 dropdowns (Q1-Q5) + 1 optional text field (Q6)
- Q5 is multi-select checkboxes
- Submit calls `POST /api/query`
- Option to click a demo persona button (calls with `{"persona": "maria"}`)

### Screen 2: Main Output (Three Panels)

**Panel A: Profile Bar** (top, full width)
- Shows `response.persona` as a human-readable label
- Shows `response.protection_gaps`

**Panel B: Priority Queue** (left column desktop, full width mobile)
- Renders `response.steps` as numbered cards
- Use `personalized_action` and `personalized_reason` if present, else `action` and `reason`
- Steps with `is_state_farm: true` get a distinct State Farm-branded card style
- Each card shows the `tier` as a subtle label
- Clicking a step should highlight its `source_nodes` in Panel C

**Panel C: Knowledge Graph** (right column desktop, below queue mobile)
- Use vis-network (vis.js) or D3.js
- `response.graph.nodes` and `response.graph.edges` are ready to render
- Node colors: purple = entry, red = risk, green = action, amber = resource
- Node size from `size` field (based on degree)
- Hover shows `title` (entity description)
- Click node: call `GET /api/node/<name>` to expand neighborhood
- Clicking a step in Panel B highlights corresponding nodes

### Screen 3: What-If Simulator
- 6 slider/toggle buttons (from `response.whatif_scenarios`)
- On click: call `POST /api/whatif` with the scenario key
- Response comes back in ~50ms
- Re-render Panel B (priority queue reorders)
- Graph emphasis updates (highlight/dim nodes) but graph structure stays same

### Graph Node Colors
```
purple = entry nodes (user's answers)
red    = risk nodes (financial dangers)
green  = action nodes (steps to take)
amber  = resource nodes (organizations, tools, hotlines)
gray   = context nodes
```
